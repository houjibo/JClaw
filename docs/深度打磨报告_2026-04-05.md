## 🎯 关键修复

### 1. 性能测试除零错误 ✅

**问题**: `ArithmeticException: / by zero`

**原因**: 性能测试执行太快（<1ms），导致 `elapsed.toMillis()` 返回 0

**修复**:
```java
Duration elapsed = Duration.between(start, Instant.now());
long elapsedMs = elapsed.toMillis();

// 避免除零错误
if (elapsedMs == 0) {
    elapsedMs = 1;  // 至少 1ms
}

long opsPerSecond = 1000 * 1000 / elapsedMs;
```

**结果**: ✅ 性能测试通过

---

### 2. Testcontainers Docker 依赖问题 ⚠️

**问题**: `Could not find a valid Docker environment`

**原因**: 
- Testcontainers 需要 Docker
- 开发环境没有 Docker

**解决方案**:

**方案 A**: 使用 H2 内存数据库（已实现）
```java
@TestConfiguration
public class IntegrationTestConfiguration {
    @Bean
    @Primary
    public DataSource dataSource() {
        return new EmbeddedDatabaseBuilder()
            .setType(EmbeddedDatabaseType.H2)
            .addScript("classpath:schema.sql")
            .build();
    }
}
```

**方案 B**: 使用 Mock 模式（推荐用于单元测试）
```java
@WebMvcTest(MemoryController.class)
class MemoryControllerTest {
    @MockBean
    private MemoryService memoryService;
}
```

**决策**: 
- 单元测试 → Mock 模式（快速、隔离）
- 集成测试 → H2 内存数据库（真实 SQL 执行）
- CI/CD 环境 → Testcontainers + Docker（最接近生产）

## 🎓 经验教训

### 1. 测试分层的重要性

**教训**: 不要所有测试都用同一模式

**最佳实践**:
- **单元测试**: Mock 所有外部依赖，只测逻辑
- **集成测试**: 真实数据库，验证 SQL 和事务
- **端到端测试**: 完整环境，验证用户流程
- **性能测试**: 生产环境配置，验证性能指标

### 2. Docker 依赖的处理

**问题**: Testcontainers 需要 Docker

**解决方案**:
1. **开发环境**: H2 内存数据库
2. **CI/CD**: Docker + Testcontainers
3. **配置切换**: 通过 Maven Profile

```xml
<profiles>
  <profile>
    <id>docker</id>
    <properties>
      <testcontainers.enabled>true</testcontainers.enabled>
    </properties>
  </profile>
</profiles>
```

### 3. 性能测试的陷阱

**陷阱**: 假设执行时间 > 0

**修复**: 防御性编程
```java
if (elapsedMs == 0) {
    elapsedMs = 1;
}
```

**最佳实践**:
- 增加迭代次数（1000 次 → 10000 次）
- 使用 JMH 专业框架
- 预热 JVM
